## 二、核心判断

### ❌ 结论：不是"移植+减枝"，是**完整重写**

| 维度 | 分析 |
|:---|:---|
| **Godot 移植版** | 几乎为空。只有 `new_script.gd`（248 字节）+ `spritemap.png`（25KB）+ 场景配置文件。没有游戏循环、没有敌人 AI、没有碰撞检测。**不能作为代码基础。** |
| **SMAPI Mod（C#）** | 有完整游戏逻辑，但深度耦合于：Stardew Valley 引擎 API、XNA/MonoGame 渲染管线、SMAPI 消息同步、Stardew 素材系统（`Game1.mouseCursors` 精灵图）。**每一行代码都假设自己在 Stardew 进程内运行。** |
| **NetLogo 版** | 教学简化实现，需要 NetLogo 解释器，不适用于浏览器环境。 |

## 三、为什么不能"移植+减枝"

### 3.1 渲染层（完全重写）

```csharp
// C# 原版（XNA/MonoGame）
b.Draw(Game1.mouseCursors, topLeftScreenCoordinate + new Vector2(position.X, position.Y), 
    new Rectangle(352 + (int)type * 16, 1696, 16, 16), 
    Color.White, 0f, Vector2.Zero, 3f, SpriteEffects.None, position.Y / 10000f + 0.001f);
```

→ 需要改写为：
```javascript
// Canvas 2D 等价
ctx.drawImage(spritesheet, 
    352 + type * 16, 1696, 16, 16,  // 源矩形
    screenX + position.x, screenY + position.y, 16 * 3, 16 * 3  // 目标矩形
);
```

**问题**：
- `Game1.mouseCursors` 是 Stardew 的 2048×2048 精灵图，受版权保护
- `SpriteEffects.None` 等 XNA 枚举需自己实现
- 深度排序（`position.Y / 10000f`）需手动管理绘制顺序
- 所有动画系统（`TemporaryAnimatedSprite`）需重写

### 3.2 输入层（完全重写）

```csharp
// C# 原版（XNA GamePad + Keyboard）
if (Game1.input.GetKeyboardState().IsKeyDown(Keys.W)) _buttonHeldState.Add(GameKeys.MoveUp);
if (pad_state.ThumbSticks.Left.X < -0.2) _buttonHeldState.Add(GameKeys.MoveLeft);
```

→ 需要改写为：
```javascript
// DOM 键盘事件
document.addEventListener('keydown', (e) => {
    if (e.key === 'w' || e.key === 'ArrowUp') keys.add('MoveUp');
});
```

### 3.3 音频层（完全重写）

```csharp
// C# 原版（Stardew 音频系统）
Game1.playSound("cowboy_gunshot");
Game1.playSound("cowboy_monsterDie");
overworldSong = Game1.soundBank.GetCue("Cowboy_OVERWORLD");
```

→ 需要：Web Audio API 或 HTML5 `<audio>` 元素，自己管理音效资源。

### 3.4 地图/瓦片系统（完全重写）

```csharp
// C# 原版（Stardew 瓦片系统）
map = MapLoader.GetMap(currentLevel);  // 从 Stardew 游戏数据加载
map[x, y] == MAP_TILE.CACTUS
map.IsCollidingWithMap(attemptedPosition)
```

→ 需要：自己设计 16×16 瓦片地图格式、碰撞层、渲染循环。

### 3.5 网络/多人同步（剥离）

```csharp
// C# 原版遍布的多人同步代码
PK_EnemyKilled message = new() { id = monster.id };
modInstance.SyncMessage(message);
```

→ 课程设计是单机游戏，所有 `SyncMessage` 调用都要删除，但这会影响代码结构。

### 3.6 存档系统（剥离+重写）

```csharp
// C# 原版（SMAPI 存档 + 二进制序列化）
modInstance.UpdateSaveState(saveState);
SaveState saveState = modInstance.GetSaveState();
```

→ 需要：`localStorage` 或 IndexedDB 的 JSON 存档。

## 四、什么可以"借鉴"（设计层面，非代码层面）

### 4.1 敌人系统（7 种敌人类型，可直接抄设计）

| 敌人类型 | 速度 | 血量 | 特性 | 掉落 |
|:---|:---:|:---:|:---|:---|
| **Orc（兽人）** | 2 | 1 | 基础近战，25% 概率"不感兴趣的"（随机游走） | 金币/道具 |
| **Ogre（食人魔）** | 1 | 3 | 慢速高血，可以踩踏 Spikey | 金币/道具 |
| **Mushroom（蘑菇）** | 3 | 2 | 快速近战 | 金币/道具 |
| **Ghost（幽灵）** | 2 | 1 | **飞行单位**（无视地形），惯性移动 | 金币/道具 |
| **Mummy（木乃伊）** | 1 | 6 | 高血慢速，25% 概率随机游走 | 金币/道具 |
| **Devil（恶魔）** | 3 | 3 | **飞行单位**，快速高血 | 金币/道具 |

### 4.3 波次/难度曲线（算法可借鉴）

```csharp
// C# 波次生成算法（简化理解）
monsterChances = [
    Vector2(0.014f, 0.4f),  // Orc 基础概率/额外概率
    Vector2.Zero,            // Ogre
    ...
];

// 每关增加概率
```

→ 概率生成算法可以移植到 JS。

### 4.4 射击系统（机制可借鉴）

- 基础射击：方向键/ WASD 决定射击方向
- 单发：单方向 1 颗子弹
- 对角线：同时按两个方向键，子弹斜向飞行
- 散射道具（SPREAD）：8 方向同时发射
- 霰弹道具（SHOTGUN）：主方向 + 两侧各 1 颗，共 3 颗
- 速射道具（RAPIDFIRE）：射击间隔 300ms → 75ms

### 4.5 升级商店（设计可借鉴）

- 弹药升级（AMMO1/2/3）：子弹伤害 +1
- 射速升级（FIRESPEED1/2/3）：射击间隔缩短
- 移速升级（RUNSPEED1/2）：移动速度提升
- 生命（LIFE）：+1 命
- 散射手枪（SPREADPISTOL）：永久散射效果
- 警长星（STAR）：一次性组合道具

## 七、推荐策略

### 方案 A：以"英雄抓怪物"为基础，升级到"草原王者 Lite"

**做法**：
1. 保留英雄抓怪物的 Canvas 框架（`game.js` 的游戏循环、键盘输入、碰撞检测）
2. 按草原王者的设计文档，逐步添加功能：
   - 第 1 阶段：加入射击系统（4 方向子弹）
   - 第 2 阶段：加入 3-4 种敌人（Orc/Ghost/Ogre）
   - 第 3 阶段：加入道具系统（心/硬币/散射/速射）
   - 第 4 阶段：加入波次生成器
   - 第 5 阶段：加入商店/升级系统
   - 第 6 阶段：加入音效、粒子特效、UI 美化

**优点**：有代码基础，渐进式开发，风险低
**缺点**：需要重构大量代码（英雄抓怪物的结构和草原王者差异很大）

### 方案 B：完全重写"草原王者无限生存模式"

**做法**：
1. 从零写 Canvas 渲染引擎
2. 阅读 C# 源码作为"设计文档"，用 JS 重新实现每个系统
3. 砍掉：关卡地图、多人模式、剧情模式、Boss 战（Dracula/Outlaw）
4. 保留：无限波次生存 + 道具 + 商店 + 7 种敌人

**优点**：代码完全自主，展示度最高，老师看不出和任何示例的关系
**缺点**：工作量大，1.5 周对新手较紧张

## 八、格雷的建议

**选方案 A（英雄抓怪物 → 草原王者 Lite）**，理由：

1. **时间安全**：1.5 周对大学生，有代码基础比从零开始安全
2. **渐进交付**：每两天可以跑通一个阶段，组员有成就感
3. **代码量可控**：在现有 380 行基础上扩展到 2000+ 行，比从零写到 2000 行快
4. **课程设计合规**：HTML/CSS/JS 纯手写，没有框架依赖，完全符合课程要求

**具体执行计划**：

| 天数 | 任务 | 负责人 | 产出 |
|:---|:---|:---|:---|
| Day 1-2 | 重构 game.js：加入玩家类、子弹类、基础射击 | 组员 A | 可射击的版本 |
| Day 2-3 | 加入 3 种敌人（Orc/Ogre/Ghost）+ AI | 组员 B | 可战斗的版本 |
| Day 3-4 | 加入道具系统（心/硬币/散射/速射） | 组员 C | 可捡道具的版本 |
| Day 4-5 | 加入波次生成器 + 难度曲线 | 组员 A/B | 无限生存版本 |
| Day 5-6 | 加入商店系统（每 2 波商人出现） | 组员 C/D | 可升级版本 |
| Day 6-7 | UI 美化（血条/得分/金币/波次 HUD）+ 音效 | 组员 D | 完整版本 |
| Day 7-9 | 测试/调参/加注释/写报告 | 全员 | 提交版本 |

## 九、素材问题

| 素材 | 来源 | 版权 | 建议 |
|:---|:---|:---:|:---|
| 角色/敌人精灵图 | Stardew Valley 原始素材 | ⚠️ 受版权保护 | 使用简单几何图形代替，或从 itch.io 找免费替代品 |
| 背景瓦片 | Stardew 原始素材 | ⚠️ 受版权保护 | 自己画简单的草地/沙漠/墓地配色块 |
| 音效 | Stardew 原始素材 | ⚠️ 受版权保护 | 从 freesound.org 找免费音效 |
| spritemap.png | Godot 移植版仓库 | ⚠️ 原始版权仍属 ConcernedApe | **不建议直接使用** |

**替代方案**：
- 玩家：绿色方块 + 帽子图标
- Orc：红色方块 + 两个白点（眼睛）
- Ghost：白色半透明方块
- Ogre：大块深红色方块
- 子弹：黄色 4×4 像素点
- 道具：彩色 8×8 方块 + 字母标识（H=心, C=硬币, S=散射）

课程设计**不要求美术质量**，简单的几何图形完全可接受。
